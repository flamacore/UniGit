# UniGit Roadmap Plan

## Plan: Native Git Client Foundation

Build a Windows-first native desktop Git client optimized for speed, one-click power workflows, and deep Unity/game-dev asset awareness. The recommended approach is a native shell with a Rust core and background worker architecture, using Git CLI plus selective native libraries for graph indexing, preview generation, and destructive-operation orchestration. The plan prioritizes a brutally fast core UX first, then layers in premium preview and advanced file/history actions without compromising frame-time or blocking the UI thread.

## Steps

1. Phase 0: Product constraints and architecture baseline. Lock Windows-first desktop as the shipping target, define the no-modal interaction model as single-click with inline preview plus undo, and set hard performance budgets for startup, frame time, graph filtering, staging refresh, and preview latency.
2. Phase 0: Establish the technical architecture. Use a native desktop shell with a Rust application core, a GPU-accelerated UI layer, asynchronous worker pools for Git and preview work, and a local cache or database layer for commit graph, preview metadata, undo records, and repo preferences.
3. Phase 1: Create the platform skeleton in parallel. Build the desktop shell, application state container, navigation model, command bus, keyboard shortcut system, drag-and-drop primitives, theme system, and telemetry or performance instrumentation. In parallel, build the repository registry, settings bootstrap, SSH discovery from the user `.ssh` directory, and process sandboxing for background tasks.
4. Phase 1: Implement the Git execution engine. Use stock Git CLI for repository operations, fetch, pull, push, commit, branch, checkout, log, show, add, reset, and apply. Wrap those commands in a cancellable task orchestration layer with structured stdout or stderr parsing, per-command timeouts, retry policies for network work, and an operation journal that records enough information to offer one-click undo where possible.
5. Phase 1: Implement the working tree and staging model. Build a dual-pane file state engine for untracked, unstaged, staged, conflicted, ignored, and locally ignored files. Support drag from unstaged to staged, drag back to unstage, context menus, hotkeys, and file badges for changed-after-staging detection by comparing index snapshots against current file system metadata and content hashes when needed.
6. Phase 1: Implement repository and remote management UX. Support add, open, remove local repositories, clone from remote, fetch, pull, push, manage multiple remotes, browse branches, tags, and worktrees, and configure repository-specific preferences with minimal setup friction. OAuth is excluded; SSH key resolution, passphrase prompting, and host verification live in the initial setup and background credential flow.
7. Phase 2: Build the commit graph and history engine. Parse and cache commit DAG data incrementally, maintain branch, tag, and ref pointers, and expose filter primitives for author, committer, message, hash, date range, branch ancestry, merge state, path, and content-sensitive search. Render the graph with row virtualization and GPU-backed drawing so very large histories remain smooth.
8. Phase 2: Build the history inspector and file-level operations. Add commit detail view, file tree per commit, file history, file restore from commit, export file from commit, revert file to commit, and file-only patch application or cherry-pick emulation. These actions depend on the Git execution engine and should record undo metadata and inline consequence summaries.
9. Phase 2: Implement destructive power commands with explicit inline guardrails. Add a visible remote action strip that includes push, plus force pull that preserves non-conflicting local state where possible, switch branch no matter what, and delete branches removed from remote. Each destructive action should avoid blocking modals, show an in-context summary of what will be discarded, create a safety ref or undo record, and execute immediately from the same surface.
10. Phase 3: Build the preview subsystem. Implement a dedicated preview coordinator that selects a renderer by file type, runs heavy decoders off the UI thread, and caches thumbnails plus extracted metadata. V1 preview scope should include fast native image preview, PSD flattened preview, FBX static mesh preview, and GLTF or GLB static preview with graceful fallback when decoding fails.
11. Phase 3: Isolate preview work for stability. Use background workers and subprocess isolation for risky or heavy decoders, enforce timeouts and memory budgets, cache preview artifacts on disk and in RAM, and never allow preview generation to block scrolling, selection, staging, or history filtering.
12. Phase 3: Add the local ignore layer. Expose a right-click local ignore action backed by repository-local exclude data rather than project `.gitignore` rules, surface ignored items in a dedicated management view, and provide restore or unignore operations.
13. Phase 4: Refine the premium UX layer. Add polished dark theme by default with light theme support, intentional motion, keyboard-first actions, alternate-click behaviors, command palette flows, hover affordances, split panes, detachable preview or history layouts, saved filters, and extremely low-friction commit or push workflows.
14. Phase 4: Harden for enterprise scale. Add crash recovery, large-repository stress handling, performance dashboards, operation tracing, repo corruption detection, long-path handling on Windows, LFS awareness, structured logs, and packaging or signing for EXE or MSI distribution.
15. Phase 5: Expand post-MVP capabilities. Add merge conflict UI, hunk-level staging, interactive rebase support, deeper PSD or FBX inspection, animation preview, Unity-specific metadata awareness, worktree management, more aggressive history queries, and cross-platform desktop readiness for Linux.

## Decisions

- Included scope: Windows-first desktop application, dark-mode-first UX, one-click actions with inline consequences plus undo, SSH-based Git auth, advanced history graph, preview pane, local ignore support, and custom power commands beyond stock Git.
- Excluded from MVP: OAuth, browser-first architecture, Android shipping target, full merge conflict editor, full PSD layer tooling, FBX animation playback, and broad cross-platform parity on day one.
- Recommended architecture: Native desktop app with Rust core and asynchronous worker or process isolation, using Git CLI as the primary execution backend and purpose-built indexing or caching for history and preview performance.
- UX rule: Ask most setup questions up front, then default to fast execution with visible inline consequences rather than modal confirmation churn.

## Verification targets

1. Confirm the architecture with a proof-of-concept that opens a large repository, renders repository status, stages files via drag-and-drop, and keeps the UI responsive during fetch, checkout, and preview generation.
2. Stress-test history rendering on repositories with very large commit graphs and validate that scroll, filter, and selection remain smooth within the defined frame budget.
3. Validate destructive operations on disposable repositories, ensuring inline previews are accurate, undo or safety refs work, and no blocking modal confirmations appear in standard workflows.
4. Validate preview generation on large PSD, FBX, GLTF, GLB, and image files, including timeout, failure, cache-hit, and crash-isolation behavior.
5. Measure startup, memory, and idle CPU against target budgets on Windows machines representative of Unity-heavy developer environments.
6. Run packaging and signing validation for EXE or MSI delivery and verify SSH-based clone, fetch, and push with common `.ssh` layouts and passphrase-protected keys.
